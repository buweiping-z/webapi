# 设备点检管理系统 — 性能优化设计（后端 + 手机端）

**日期**: 2026-06-15
**状态**: 待实施
**优化级别**: B（适中 — 不改表结构，保持 API 契约）
**覆盖范围**: ASP.NET Core 后端 + Android (Jetpack Compose) 手机端

---

## 概述

当 inspection_records / inspection_results 增长到 10-100 万级时，后端代码存在全表扫描、N+1 查询、无分页、变更跟踪开销等性能瓶颈。同时 Android 手机端存在 API 调用无防抖、资源泄漏、缓存缺失等问题。项目处于开发中阶段，两端可同步优化。

## 后端问题清单

### 1. 缺失索引 → 全表扫描

| # | 表 | 缺失列 | 影响 API |
|---|-----|-------|---------|
| 1 | inspection_results | record_id | 所有 record↔result 的 JOIN 查询 |
| 2 | inspection_results | (record_id, item_name) | SaveDailyRecord 的 upsert 匹配 |
| 3 | inspection_records | (device_model, inspection_time) | GetMonthlyRecords, GetDailyOperators, GetMonthlySummary, BuildFrequencyStats |
| 4 | inspection_signatures | (device_model, year, month) | GetSignatures, SaveSignatures, AutoSign |

### 2. N+1 查询

- **BuildFrequencyStats**（InspectionController.cs:763-795）：对每个 device 循环执行 3 次数据库查询。100 台设备 = 300 次 SQL。
- **SaveDailyRecord**（InspectionController.cs:196-264）：对每条 item 逐条查 records → 逐条查 results → 逐条 insert/update。

### 3. 无分页 + 变更跟踪开销

- GetMonthlyRecords 返回某设备整月全量数据
- 所有只读 GET 端点未使用 AsNoTracking()
- 大部分查询使用 SELECT * 拉取全部列

### 4. 低效数据分组

- GetDailyOperators：加载全月 records 到内存再做 GroupBy
- GetMonthlyRecords：值转换（正常→○）在内存 foreach 中执行

## 优化方案

### P0：数据库索引（EF Core Migration）

新增 3 个索引。关键区别：`inspection_results(record_id, item_name)` 使用 **UNIQUE 索引**，这是后续 `INSERT ON DUPLICATE KEY UPDATE` 批量 upsert 的前提。

**前置条件**：执行迁移前需确认 `inspection_results` 表中没有重复的 `(record_id, item_name)` 数据。如有，先清理。

```csharp
// AppDbContext.OnModelCreating

// 1. inspection_results — 唯一索引，支持批量 upsert
entity.HasIndex(r => new { r.RecordId, r.ItemName })
    .IsUnique()
    .HasDatabaseName("uq_results_record_item");

// 2. inspection_records — 月度时间范围查询
entity.HasIndex(r => new { r.DeviceModel, r.InspectionTime })
    .HasDatabaseName("idx_records_device_time");

// 3. inspection_signatures — 自然键按月查询
entity.HasIndex(s => new { s.DeviceModel, s.Year, s.Month })
    .HasDatabaseName("idx_signatures_device_period");
```

生成迁移：`dotnet ef migrations add AddPerformanceIndexes`

### P0：全局 AsNoTracking

在 `AppDbContext` 构造函数中一处配置，覆盖所有查询。比每个端点手写 `.AsNoTracking()` 更干净，不会遗漏。

```csharp
public AppDbContext(DbContextOptions<AppDbContext> options) : base(options)
{
    ChangeTracker.QueryTrackingBehavior = QueryTrackingBehavior.NoTracking;
}
```

**需要更新/删除的查询**显式使用 `.AsTracking()`。当前只需要在以下位置加：
- `SaveDailyRecord` 中查找已有 record 的查询
- `SubmitFullInspection` 中删除已有 record 的查询
- `SaveSignatures` 中查找已有 signature 的查询
- `AutoSign` 中查找已有 signature 的查询
- `AddOperators` 中查重 qualified_inspectors 的查询（如有）
- `DeleteOperator` 中查找 qualified_inspectors 的查询

### P0：BuildFrequencyStats 批量化

将 foreach device 循环内的 3 次独立查询合并为 3 次全量查询 + 内存分组：

```
Before（N devices = 3N 次 SQL）:
  for each device:
    SELECT ... FROM templates WHERE device_model = ?
    SELECT ... FROM records WHERE device_model = ?
    SELECT ... FROM results WHERE record_id IN (...)

After（固定 3 次 SQL）:
  SELECT ... FROM templates WHERE device_model IN (all)
  SELECT ... FROM records WHERE device_model IN (all) AND time range
  SELECT ... FROM results WHERE record_id IN (all)
  内存中按 device 维度分组匹配
```

防护措施：
- `recordIds` 为空时直接短路返回空结果（避免 `IN ()` 语法错误）
- `deviceModels` 数量 > 500 时分批处理 IN 子句

### P0：SaveDailyRecord 批量 Upsert

前提：`uq_results_record_item` 唯一索引已创建。

方案：
1. 确保 inspection_record 存在（查一次 + 必要时 insert）
2. 用事务包裹
3. 用 `INSERT ... ON DUPLICATE KEY UPDATE` 批量写入 results

使用 EF Core 的 `ExecuteSqlRawAsync`，事务保证原子性：

```csharp
using var transaction = await _context.Database.BeginTransactionAsync();
try
{
    var recordId = await GetOrCreateRecord(request);
    
    var sql = @"
        INSERT INTO inspection_results (record_id, item_name, result_value, is_normal, remark)
        VALUES ({0}, {1}, {2}, {3}, {4})
        ON DUPLICATE KEY UPDATE
            result_value = VALUES(result_value),
            is_normal = VALUES(is_normal),
            remark = VALUES(remark)";
    
    foreach (var item in request.Items)
    {
        await _context.Database.ExecuteSqlRawAsync(sql,
            recordId, item.ItemName, item.ResultValue, item.IsNormal, item.Remark);
    }
    
    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
```

### P1：分页标准化

创建通用分页响应 DTO：

```csharp
public class PagedResponse<T>
{
    public List<T> Items { get; set; } = new();
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalCount { get; set; }
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
}
```

改造端点（向后兼容，不传参时默认 pageSize=100）：

| 端点 | 新增参数 | 默认值 |
|------|---------|--------|
| GetMonthlyRecords | page, pageSize | page=1, pageSize=100 |
| GetUserList | page, pageSize | page=1, pageSize=100 |
| GetOperatorList | page, pageSize | page=1, pageSize=100 |

必须配合稳定排序 `ORDER BY inspection_time DESC, id DESC` 确保分页结果不重复/不遗漏。

### P1：DTO 投影优化

所有只读接口改为 `.Select()` 投影返回 DTO，避免 SELECT *：

```csharp
// 定义 DTO（放在 Controllers 目录或 Models 目录）
public class DeviceDto
{
    public string DeviceModel { get; set; } = "";
    public string DeviceName { get; set; } = "";
}

// Controller 返回 DTO，EF Core 自动生成最小列查询
public async Task<List<DeviceDto>> GetDevices()
{
    return await _context.Devices
        .Select(d => new DeviceDto
        {
            DeviceModel = d.DeviceModel,
            DeviceName = d.DeviceName
        })
        .ToListAsync();
}
```

### P1：GetDailyOperators 窗口函数优化

使用 MySQL 8.0+ 的 `ROW_NUMBER()` 窗口函数替代内存 GroupBy（需先确认数据库版本 `SELECT VERSION()`）：

```csharp
var sql = @"
    SELECT day, employee_id FROM (
        SELECT 
            DAY(inspection_time) as day,
            employee_id,
            ROW_NUMBER() OVER (PARTITION BY DAY(inspection_time) ORDER BY inspection_time DESC) as rn
        FROM inspection_records
        WHERE device_model = {0} 
            AND inspection_time >= {1} 
            AND inspection_time <= {2}
    ) t WHERE rn = 1";

return await _context.Database
    .SqlQueryRaw<DailyOperatorResult>(sql, deviceModel, startDate, endDate)
    .ToListAsync();
```

### 不做的事情

- ❌ SQL CASE WHEN 值转换 — 审查认为高风险，保持内存转换
- ❌ 引入 Redis/缓存层 — B 级优化不涉及
- ❌ 改表结构（新增列等）— 本次不涉及
- ❌ AutoDetectChanges 修改 — 审查不建议
- ❌ P2 软删除 + 审计字段 — 属于 C 级改动，标记为未来评估

## 后端兼容性保证

- 所有 API 端点路由不变
- 请求/响应 JSON 结构向后兼容（分页端点新增分页元数据，原有字段不变）
- 数据库变更仅新增索引，不修改表结构
- 所有变更通过 `dotnet ef database update` 一键应用
- 换一台电脑：`dotnet restore && dotnet ef database update && dotnet run`
